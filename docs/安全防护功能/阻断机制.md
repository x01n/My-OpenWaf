## 简介
本文件系统性阐述 My-OpenWaf 的阻断机制，覆盖阻断决策的触发条件与执行流程、阻断类型与响应头设置、错误页面定制、阻断策略配置（持续时间、重定向、自定义消息）、阻断日志与审计、阻断与缓存系统的交互、阻断测试与验证方法以及性能影响与优化策略。特别关注新增的零响应阻断机制和详细的阻断事件追踪功能。

## 项目结构
阻断机制由"决策引擎 + 执行管线 + 响应渲染 + 日志审计 + 缓存隔离 + 阻断执行器"构成，前后端共同提供可定制的阻断页面与诊断信息。

## 核心组件
- 决策与动作
  - 动作类型与终端语义：定义允许、拦截、观察、丢弃等动作及是否短路、是否记录日志等判定逻辑。
  - 终止条件：拦截与丢弃为终端动作，会短路后续阶段与上游请求。
- 处理引擎与管线
  - 引擎按顺序组装多阶段规则（IP信誉、ACL、机器人检测、速率限制、OWASP/CVE签名、自定义规则）。
  - 管线逐阶段执行，遇到终端结果立即返回。
- 阻断执行器
  - 零响应阻断：直接关闭TCP连接，不发送任何HTTP响应，适用于高风险攻击场景。
  - 统计追踪：原子计数器记录各类阻断来源和时间戳。
  - 原子操作：使用原子类型确保并发安全性。
- 阻断响应渲染
  - 支持站点级/全局级自定义 HTML 拦截页与状态码；若无模板则回退到内置嵌入页面。
  - 设置标准化响应头（如请求 ID、WAF 动作类型），便于审计与追踪。
- 阻断策略管理
  - 全局策略配置：启用/禁用、Bot分数阈值、CVE自动Drop设置。
  - 实时统计：24小时阻断统计和来源分布。
  - 事件追踪：详细的阻断事件列表和过滤功能。
- 日志与审计
  - 观察命中事件异步批量写入数据库，不阻塞热路径。
  - 访问日志与安全事件日志统一记录请求 ID、规则 ID、匹配描述、分类等。
- 缓存隔离
  - 阻断请求不进入上游，避免污染响应缓存；响应缓存仅针对安全 GET 请求进行缓存。

## 详细组件分析

### 阻断类型与触发条件
- 动作类型
  - 允许：放行请求。
  - 拦截：终端动作，阻断并返回阻断页。
  - 观察：非终端动作，仅记录日志，不影响放行。
  - 丢弃：最高优先级终端动作，直接关闭 TCP 连接，不发送任何响应。
- 触发条件
  - 引擎按序执行各阶段规则，任一阶段匹配且动作为拦截或丢弃即短路。
  - 机器人检测在高风险情况下可直接触发丢弃。
  - IP 信誉白名单可短路放行，黑名单可直接触发拦截。
  - 速率限制超过阈值可触发拦截或丢弃（取决于配置）。

**更新** 新增零响应阻断机制，适用于高风险攻击场景，提供更彻底的阻断效果。

- 执行器设计

### 阻断机制与缓存系统的交互
- 缓存隔离原则
  - 阻断请求不进入上游，因此不会污染响应缓存。
  - 响应缓存仅对安全的 GET 请求进行缓存，命中后直接返回缓存内容。
- 缓存键与过期
  - 缓存键由方法、主机、路径、查询串确定，过期时间基于 TTL 判断。
  - 缓存启停、默认 TTL、统计与清理均内置管理。
- 避免缓存污染
  - 阻断场景不写入缓存；缓存仅对允许的 GET 响应生效。

## 性能考虑
- 热路径优化
  - 观察命中采用异步批量写入，避免阻塞请求处理。
  - 管线短路机制减少无效计算，终端动作立即返回。
  - 零响应阻断直接关闭TCP连接，避免HTTP处理开销。
- 缓存策略
  - 响应缓存采用分片锁与定期清理，降低锁竞争与内存占用。
  - 对单条目过大进行保护，避免缓存被异常大响应撑爆。
- 速率限制与 IP 信誉
  - 固定窗口计数与自动封禁窗口均带清理任务，防止长期占用内存。
- 阻断执行器性能
  - 原子计数器确保统计操作的高性能。
  - 零响应阻断避免HTTP响应生成的额外开销。
- 建议
  - 合理设置速率限制窗口与阈值，避免误伤正常用户。
  - 控制自定义拦截 HTML 大小，避免模板解析与传输开销。
  - 在高并发场景下，适当增大事件写入缓冲区与批量大小。
  - 合理配置阻断策略，避免过度阻断影响用户体验。

## 故障排除指南
- 阻断页面未显示或显示默认页
  - 检查站点级/全局拦截 HTML 是否配置正确，模板语法是否有效。
  - 若模板解析失败，将回退到内置 HTML，确认日志中是否有解析错误记录。
- 请求 ID 未出现在页面
  - 确认阻断渲染是否设置了 X-Request-ID 头，前端占位符是否被正确替换。
- 观察命中未入库
  - 检查事件写入器缓冲区是否溢出（日志中可能有丢弃警告），适当增大缓冲或批量大小。
- 误封禁或漏封禁
  - 核对 IP 黑/白名单与自动封禁阈值、窗口、持续时间配置。
- 速率限制误判
  - 调整窗口秒数与最大请求数，或临时禁用以定位问题。
- 零响应阻断问题
  - 检查阻断执行器是否启用，确认TCP连接是否正确关闭。
  - 验证阻断统计是否正常更新。
  - 确认阻断事件是否正确记录到数据库。
- 阻断策略配置问题
  - 检查阻断策略API接口是否正常工作。
  - 验证前端阻断策略界面的数据加载和保存功能。

## 结论
My-OpenWaf 的阻断机制通过"引擎 + 管线 + 动作语义 + 阻断执行器"的清晰分层，实现了可配置、可观测、可扩展的阻断能力。其核心特性包括：
- 明确的动作优先级与短路机制，确保拦截与丢弃的即时生效；
- 可定制的阻断页面与标准化响应头，便于审计与用户提示；
- 零响应阻断机制，提供更彻底的阻断效果，适用于高风险攻击场景；
- 详细的阻断事件追踪与统计分析，支持实时监控和决策优化；
- 异步日志与缓存隔离，保障性能与一致性；
- 丰富的配置项（速率限制、IP 信誉、自动封禁、阻断策略）满足多样化防护需求。

新增的零响应阻断机制和阻断策略管理功能进一步增强了系统的安全性和可管理性。建议在生产环境中结合业务流量特征合理配置阈值与页面，配合日志与监控体系进行持续优化。
